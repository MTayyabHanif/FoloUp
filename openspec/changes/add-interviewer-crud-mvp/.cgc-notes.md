## 5. Migration SQL (Sketch)

```sql
-- Step 1: Add prompt column with temporary default for existing rows
ALTER TABLE interviewer
  ADD COLUMN prompt TEXT NOT NULL DEFAULT '';

-- Step 2: Backfill existing rows by name
-- Lisa and Bob share the general prompt; Robust Bot has its own.
UPDATE interviewer
SET prompt = 'You are an interviewer who is an expert in asking follow up questions...'
-- (full RETELL_AGENT_GENERAL_PROMPT text)
WHERE name IN ('Explorer Lisa', 'Empathetic Bob');

UPDATE interviewer
SET prompt = '# Role\n\nYou are conducting a first-round screening interview...'
-- (full RETELL_AGENT_ROBUST_BOT_PROMPT text)
WHERE name = 'Robust Bot';

-- Step 3: Drop the default now that all rows have real values
ALTER TABLE interviewer
  ALTER COLUMN prompt DROP DEFAULT;

-- Step 4: Add soft-delete column
ALTER TABLE interviewer
  ADD COLUMN deleted_at TIMESTAMPTZ NULL;

-- Step 5: Add voice_id column (needed for Decision 4 ‚Äî CRUD create must persist the chosen voice)
ALTER TABLE interviewer
  ADD COLUMN voice_id TEXT NULL;

-- Step 6: Backfill voice_id for existing rows
UPDATE interviewer SET voice_id = '11labs-Chloe' WHERE name = 'Explorer Lisa';
UPDATE interviewer SET voice_id = '11labs-Brian' WHERE name IN ('Empathetic Bob', 'Robust Bot');
```

Note: The actual prompt text must be copy-pasted verbatim from `src/lib/constants.ts` in the migration. The backfill relies on interviewer name matching exactly ‚Äî if duplicate names exist (idempotency gap), all rows with that name get the same prompt, which is correct.

Also required: regenerate or hand-update `src/types/database.types.ts` after migration.

## 6. Open Architectural Questions for Propose

**Q1 ‚Äî API route structure:** Should the new endpoints live at `/api/interviewers` (REST) or follow the existing one-off pattern (e.g., `/api/delete-interviewer`)? The existing routes are not RESTful (they're all flat). A REST approach (`/api/interviewers` + `/api/interviewers/[id]`) is cleaner but diverges from the current pattern.

**Q2 ‚Äî Context vs react-query for mutations:** `useInterviewersQuery` exists but is unused. The existing context uses full refetch. Should the CRUD MVP wire up the existing hook (enabling `useInvalidateInterviewers` for clean post-mutation refresh) or stick with the context's `fetchInterviewers()` pattern? Wiring the hook is better long-term but touches more files.

**Q3 ‚Äî Create form: new page or modal?** The dashboard shows a card-grid. A "New Interviewer" card (like the old bootstrap button) opening a modal is the least-disruption path. A dedicated `/dashboard/interviewers/new` page is cleaner for a form with a textarea. Brainstorming didn't decide.

**Q4 ‚Äî Delete confirmation:** Should delete be a single-click (trash icon ‚Üí immediately soft-delete) or require a confirmation dialog? Given that `interview` rows FK-reference `interviewer.id`, soft-deleting an interviewer that has active interviews attached needs consideration (the `interview` row keeps `interviewer_id` pointing to a soft-deleted row, which is fine for `getInterviewer` since it bypasses `deleted_at`, but the interviewer picker in the create-interview flow must not show deleted interviewers).

**Q5 ‚Äî `create-interviewer` route fate:** Keep as deprecated seed route, redirect to new POST, or remove entirely? Brainstorming deferred to propose. Given the idempotency check in `createInterviewer` (name + agent_id), re-running the seed route after CRUD interviewers are created won't duplicate them ‚Äî but it won't know about user-created agents. Propose should decide.

**Q6 ‚Äî Retell LLM creation on CRUD create:** The current seed route creates a Retell LLM object per distinct prompt, then a Retell agent object per interviewer. The new CRUD `POST /api/interviewers` must do the same ‚Äî but is it one LLM per interviewer (one LLM per unique prompt string) or one LLM per interviewer always? For simplicity, one LLM per interviewer (even if prompts are identical) avoids sharing state. Propose should decide.

**Q7 ‚Äî `interviewerDetailsModal` prompt display:** The detail modal currently doesn't show the prompt. Should the CRUD MVP add a prompt read-only view to the modal? This is a natural companion to creation but brainstorming didn't mention it.
