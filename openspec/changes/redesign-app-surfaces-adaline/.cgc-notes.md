## 2. Relationship Map

### Provider and shell dependencies

- `src/components/providers.tsx` wraps every app surface in:
  - `NextThemesProvider`
  - `QueryClientProvider`
  - `InterviewProvider`
  - `InterviewerProvider`
  - `ResponseProvider`
  - `ClientProvider`
- `src/app/(client)/layout.tsx` composes `Providers` with `AppSidebar`, `AppHeader`, `Drawer`, `AppFooter`, and route-group toasts.
- `src/app/(user)/layout.tsx` composes `Providers` with `CandidateFooter` and candidate toasts.
- `src/components/shell/app-sidebar.tsx` is the only in-scope shell component still directly coupled to `next-themes` via `useTheme()`.

### Interview data flow

- `src/contexts/interviews.context.tsx`
  - Fetches the recruiter interview list via `InterviewService.getAllInterviews(userId, organizationId)`.
  - Exposes `getInterviewById()` to both recruiter and candidate route groups.
- Call sites for interview data:
  - `src/app/(client)/dashboard/page.tsx` reads `useInterviews()` for the dashboard inventory.
  - `src/app/(client)/interviews/[interviewId]/page.tsx` reads `getInterviewById()` to hydrate the job workspace.
  - `src/app/(user)/call/[interviewId]/page.tsx` reads `getInterviewById()` to validate and hydrate the candidate flow.
  - `src/components/call/callInfo.tsx` reads `InterviewService.getInterviewById()` again to resolve `question_count` for coverage.

### Response data flow

- `src/services/responses.service.ts`
  - `getAllResponses(interviewId)` is the main source for recruiter pipeline/session state.
  - `createResponse(payload)` is the candidate-side insert path when a new session starts.
  - `updateResponse()` / `saveResponse()` drive candidate-status changes, session-token reassignment on reconnect, and "viewed" flags.
  - `getAllEmails(interviewId)` gates duplicate candidate entry before a new call starts.
- Call sites for response data:
  - `src/app/(client)/dashboard/page.tsx` uses `getResponseCountByOrganizationId()` for free-plan quota handling.
  - `src/components/dashboard/interview/interviewCard.tsx` uses `getAllResponses()` per card, then pings `/api/get-call` for each unanalysed response.
  - `src/app/(client)/interviews/[interviewId]/page.tsx` uses `getAllResponses()` to build the response rail and summary/detail panes.
  - `src/components/call/index.tsx` uses `createResponse()`, `updateResponse()`, and `getAllEmails()`.
  - `src/components/call/callInfo.tsx` uses `getResponseByCallId()`, `updateResponse()`, and `deleteResponse()`.

### Interviewer data flow

- `src/contexts/interviewers.context.tsx`
  - Fetches the shared interviewer roster via `InterviewerService.getAllInterviewers(userId)`.
  - Exposes `deleteInterviewer()` and `fetchInterviewers()`.
- Call sites for interviewer data:
  - `src/app/(client)/dashboard/interviewers/page.tsx` renders the interviewer library from context.
  - `src/components/dashboard/interviewer/CreateInterviewerModal.tsx` posts to `/api/interviewers` then calls `fetchInterviewers()`.
  - `src/components/dashboard/interviewer/InterviewerCard.tsx` uses `deleteInterviewer()`.
  - `src/components/dashboard/interview/summaryInfo.tsx`, `create-popup/details.tsx`, and `editInterview.tsx` all depend on the same roster for interviewer identity and selection.
  - `src/components/dashboard/interview/interviewCard.tsx` and `src/components/call/index.tsx` each make direct `InterviewerService.getInterviewer()` lookups for image/presence data.

### Candidate session lifecycle dependencies

- `src/components/call/index.tsx` directly orchestrates:
  - `POST /api/register-call`
  - `GET /api/check-session?token=...`
  - `PATCH /api/response-heartbeat`
  - `InterviewerService.getInterviewer()`
  - `ResponseService.getAllEmails()`
  - `ResponseService.createResponse()`
  - `ResponseService.updateResponse()`
- Derived workflow already present in data:
  - Candidate pipeline state comes from `response.status` (`ongoing`, `completed`, `interrupted`, `abandoned`).
  - Recruiter outcome state comes from `response.candidate_status`.
  - Session integrity/quality signals come from `is_viewed`, `disconnection_reason`, `questions_covered`, `tab_switch_count`, and `analytics`.

## 3. Change Footprint Per Locked Decision

### Decision 1 — hiring-workflow IA

- Primary files:
  - `src/app/(client)/dashboard/page.tsx`
  - `src/components/dashboard/interview/interviewCard.tsx`
  - `src/app/(client)/interviews/[interviewId]/page.tsx`
  - `src/components/dashboard/interview/summaryInfo.tsx`
  - `src/components/call/callInfo.tsx`
  - `src/components/ui/page-shell.tsx`
- Reason:
  - The data already supports a workflow-first framing.
  - `interview` is the job definition.
  - `response.status` and `response.candidate_status` already power a derived pipeline.
  - The redesign is mostly a view-model and layout problem, not a backend problem.

### Decision 2 — jobs/pipeline is a derived UX layer

- Existing fields already available for the abstraction:
  - Job definition: `interview.name`, `objective`, `description`, `questions`, `interviewer_id`, `time_duration`
  - Pipeline/session state: `response.status`, `candidate_status`, `analytics`, `created_at`, `questions_covered`, `disconnection_reason`
- No DB or route changes are required to introduce:
  - Job cards on `/dashboard`
  - Stage summaries on `/interviews/[interviewId]`
  - Candidate rails grouped by live/interrupted/abandoned/reviewed

### Decision 3 — Adaline light-only theme

- Hard change points:
  - `src/app/globals.css`: replace ADS/shadcn token mix and remove both `.dark` blocks.
  - `src/components/providers.tsx`: remove `NextThemesProvider`.
  - `src/components/shell/app-sidebar.tsx`: remove `SidebarThemeToggle()` and `useTheme()`.
  - Shared primitives under `src/components/ui/`: remap radius, borders, shadows, focus rings, and default fills.
- Scope note:
  - Dark-specific utility classes are still present inside the candidate call runtime (`src/components/call/index.tsx`), not just the token layer.

### Decision 4 — candidate flow should feel calm and guided

- Primary files:
  - `src/app/(user)/call/[interviewId]/page.tsx`
  - `src/components/call/index.tsx`
  - `src/components/call/callInfo.tsx`
  - `src/components/call/feedbackForm.tsx`
  - `src/components/call/tabSwitchPrevention.tsx`
- Existing structure helps:
  - Candidate route already has dedicated invalid/closed/loading/mobile shells.
  - Reconnect phases are already explicit (`checking`, `ready_to_resume`, `expired`, `not_found`, `offline`, `check_failed`, `register_failed`).
- Existing conflict:
  - `interview.theme_color` still drives prominent candidate UI accents and would cut against a consistent system if left untouched.

### Decision 5 — dashboard answers "what needs attention?"

- Primary files:
  - `src/app/(client)/dashboard/page.tsx`
  - `src/components/dashboard/interview/interviewCard.tsx`
  - `src/components/ui/banner.tsx`
  - `src/components/ui/empty-state.tsx`
- Existing blockers:
  - Dashboard state is mostly inventory plus quota gating.
  - Interview cards currently hide the richer session-state signals behind per-card fetching and a very simple count UI.

### Decision 6 — interview detail becomes the hiring workspace

- Primary files:
  - `src/app/(client)/interviews/[interviewId]/page.tsx`
  - `src/components/dashboard/interview/summaryInfo.tsx`
  - `src/components/call/callInfo.tsx`

- Existing foundation:
  - The route already uses a master-detail pattern.

### Decision 7 — interviewer management becomes a persona library

- Primary files:
  - `src/app/(client)/dashboard/interviewers/page.tsx`
  - `src/components/dashboard/interviewer/InterviewerCard.tsx`

## 4. Risks and Constraints

### R1 — `src/components/call/index.tsx` is the highest-risk file in scope

- It mixes transport logic, reconnect logic, session persistence, timers, mic permission handling, heartbeat flushing, transcript rendering, and multiple candidate screens in one client component.
- Visual redesign alone is possible, but a clean result will likely require splitting the file into smaller view/state units during `apply`.

### R2 — Theme removal is deeper than a token swap

- `next-themes` is wired at the provider layer.
- `app-sidebar.tsx` exposes a theme toggle.
- `globals.css` defines two `.dark` blocks.
- `src/components/call/index.tsx` still contains explicit `dark:` utility classes and theme-rotation icon states.

### R3 — Current dashboard cards are network-heavy and not pipeline-ready

- `src/components/dashboard/interview/interviewCard.tsx` performs:
  - one interviewer fetch per card
  - one responses fetch per card
  - one `/api/get-call` call per unanalysed response
- A more decision-oriented dashboard will amplify this pain unless the redesign also rationalizes where counts and status summaries are derived.

### R4 — `theme_color` conflicts with the locked single-theme direction

- Recruiter detail page allows editing `interview.theme_color`.
- Candidate flow uses `interview.theme_color` for prominent buttons and borders.
- Propose should decide whether to:
  - demote it to a subtle marker,
  - constrain it to a tiny curated accent usage,
  - or remove the UI affordance while leaving stored data untouched.

### R5 — CGC coverage for this repo is partial

- `cgc stats` reports 2 repositories in the active graph.
- Symbol/dependency queries for the current TypeScript page/component names returned no matches or no dependency info, so the graph is not trustworthy for fine-grained TS call graphs here.
- The relationship map above was therefore verified by direct source reads plus call-site searches (`rg`) against the scoped files.
- This is not a blocker for the redesign, but it means `propose` should treat these notes as the authoritative dependency map for this change.

## 5. Proposed Implementation Seams for `propose`

- Build the redesign around three top-level surface systems:
  - recruiter shell/system
  - job workspace system
  - candidate session system
- Treat `src/components/ui/` plus `src/app/globals.css` as the first-wave system rewrite, then relayout pages on top of it.
- Preserve current data contracts and route structure where possible:
  - `/dashboard` becomes the command center
  - `/interviews/[interviewId]` becomes the job workspace
  - `/dashboard/interviewers` becomes the persona library
  - `/call/[interviewId]` remains the candidate entry route
- Expect `apply` to touch both layout primitives and feature components; the redesign cannot be done cleanly by editing page files alone.

## 6. Open Questions for `propose`

- Should the dashboard stay as one dense command center page, or split into clearly stacked zones such as live activity, hiring pipeline, and job library while still preserving a single route?
- Should the interview detail page keep inline edit mode (`?edit=true`) inside the workspace, or move configuration into a secondary drawer/panel so candidate review stays primary?
- Should the old per-interview theme color remain as a tiny badge/accent only, or should the recruiter-facing color picker be removed from the workspace entirely?
- Should the interviewer create flow remain modal-based, or graduate to a richer side-panel/page treatment if the editorial storytelling becomes too constrained inside the current modal?
