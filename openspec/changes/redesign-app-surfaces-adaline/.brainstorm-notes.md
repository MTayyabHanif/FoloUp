# Brainstorm notes — redesign-app-surfaces-adaline

Locked decisions from the `/flow` brainstorming step. The `cgc`, `propose`,
and review steps should treat these as inputs, not reopen them unless a later
plan review finds a concrete implementation risk.

## Decisions

### 1. Information architecture: shift from feature buckets to hiring workflow
**Decision:** Reframe the app around the hiring workflow, not around isolated feature lists.

**Implication:**
- `/dashboard` becomes a recruiter command center with live session visibility, stage counts, job/interview health, and clear next actions.
- `/interviews/[interviewId]` becomes the primary "job workspace" even though the backend entity remains `interview`.
- `/dashboard/interviewers` becomes a curated interviewer library/studio, not just a simple card grid.
- `/call/[interviewId]` remains the candidate entry route, but the experience is rebuilt as a calmer guided session flow.

### 2. Jobs are a derived UX layer, not a new database concept
**Decision:** Introduce a jobs/pipeline mental model only as a presentation layer derived from existing interview and response data.

**Implication:**
- No new `job` table, route contract, or migration is required.
- Existing interview fields (`name`, `objective`, `description`, `questions`, `interviewer_id`, `time_duration`) become the job definition.
- Existing response fields (`status`, `candidate_status`, `analytics`, `created_at`, `questions_covered`, `disconnection_reason`) power the candidate pipeline and session health views.
- Any new UI copy can say "job" where that improves comprehension, but code and persistence continue to use the current interview model.

### 3. Theme system: full Adaline adoption, single light mode only
**Decision:** Replace the current Atlassian/shadcn-dark-mode mix with one Adaline-based light theme system across recruiter and candidate surfaces.

**Implication:**
- `src/app/globals.css` becomes the source of truth for Adaline color, spacing, radius, and typography tokens.
- Remove `.dark` token blocks, `next-themes` provider usage, the sidebar theme toggle, and dark-mode-specific utility classes.
- Replace the current blue/purple brand ramp and heavier shadows with muted canvas surfaces, restrained borders, and the Adaline green/brown accent logic.
- UI components should be normalized onto the same surface language instead of mixing old shadcn defaults, slate panels, and one-off status colors.

### 4. Candidate flow should feel calm, trustworthy, and guided
**Decision:** The candidate experience should prioritize clarity, confidence, and low-friction readiness over dashboard-style chrome.

**Implication:**
- The preflight state should clearly communicate role context, time expectation, microphone readiness, and what will happen next.
- Reconnect, expired-session, invalid-link, closed-interview, and mobile-blocked states should all use the same calm visual system instead of ad hoc error cards.
- The in-call surface should emphasize focus, progress, interviewer presence, and a sense of being guided through one session, not dropped into a generic widget.
- The current per-interview `theme_color` should not drive broad candidate-page theming; at most it may survive as a small accent marker if it does not break system consistency.

### 5. Recruiter dashboard should surface decisions and bottlenecks, not just inventory
**Decision:** The main dashboard should answer "what needs attention?" before it answers "what exists?"

**Implication:**
- Prioritize live sessions, candidates needing review, selected/potential candidates, interrupted/abandoned sessions, and recent hiring activity near the top.
- Interview cards should evolve into more useful job cards with candidate counts, session health, and readiness signals rather than decorative thumbnails alone.
- Empty states should feel intentional and directional, with clear CTAs and context, not just placeholders.
- The free-plan/quota state should be redesigned into the same system rather than feeling like a separate legacy modal.

### 6. Interview detail should become a hiring workspace
**Decision:** Redesign the interview detail page as the place where recruiters review candidates, interpret signals, and act.

**Implication:**
- Keep the existing route and API shape, but relayout the page around a strong workspace hierarchy: overview + candidate pipeline + candidate detail.
- The response list should behave like a pipeline/session rail, with better grouping of live, interrupted, abandoned, and reviewed candidates.
- The summary view should emphasize hiring signal, candidate status distribution, completion quality, and standout candidates before low-level analytics detail.
- Individual response detail should foreground recommendation, overall signal, transcript/recording, session integrity, and candidate-status actions.

### 7. Interviewer management should feel like a persona library, not a utility screen
**Decision:** Rebuild the interviewer area as a coherent library of interviewer personas with stronger storytelling and better create/manage affordances.

**Implication:**
- Interviewer cards should communicate persona identity, purpose, and conversation style more clearly.
- The create flow should read as composing a persona, not filling a raw admin form.
- Voice, prompt, and conversation traits should remain visible, but the presentation should feel editorial and productized rather than mechanical.
- Existing CRUD behavior and current data fields remain sufficient; this is a UX restructure, not a capability expansion.

### 8. Typography fidelity is desired, but font assets are not in the repo
**Decision:** Wire the Adaline typography tokens now, but do not block the redesign on missing local font files.

**Implication:**
- Define `--font-akkurat` and `--font-fragmentmono` tokens in the system layer and apply them consistently.
- If the repo still lacks self-hosted Akkurat / Fragment Mono assets at apply time, use named-family stacks with graceful fallbacks while preserving spacing, sizing, and case rules.
- Exact font fidelity is a polish dependency, not a reason to delay the structural redesign.

## Non-goals for this change

- Auth, homepage, marketing, or static page redesign
- Database migrations or new backend entities
- A true first-class jobs data model
- Rewriting the candidate interview engine or response lifecycle semantics
- Preserving dark mode as an option
- Visual one-off exceptions that bypass the Adaline system

## Open questions deferred to propose / plan reviews

- Whether a dedicated jobs route is worth adding, or whether the jobs-first UX should stay anchored on the existing `/dashboard` and `/interviews/[id]` routes
- How aggressively to simplify or remove the current per-interview theme-color customization
- Which existing shared components should be rebuilt outright versus reskinned in place for the fastest clean result
